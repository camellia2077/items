# Character Switch Strategy

## 背景

在 Breach 角色页中，`Robot` 的“点击后持久解锁”在部分环境下表现不稳定：

* 点击后提示已解锁，但重启游戏后仍未真正可选。
* 同一套 `GungeonFlags` 写入逻辑在不同环境下表现不一致，存在“看起来成功、实际不可用”的情况。

这个问题直接影响玩家可用性：用户目标是“现在切到 Robot”，而不是“将来某次可能解锁成功”。

## 当前策略

对于 `Robot`，当前策略是：

* 不把“持久解锁”作为主路径。
* 在 Breach 内点击 `Robot` 时，直接执行强制角色切换。

换句话说，`Robot` 的按钮语义是“立即可用地切换角色”，而不是“尝试修改存档解锁状态”。

## 为什么优先强制切换

* 可用性优先：用户操作后立刻得到可验证结果。
* 风险可控：避免把功能正确性绑定到不稳定的存档解锁链路。
* 调试成本更低：强制切换路径更容易复现、验证和回归测试。

## 取舍

该策略的明确取舍是：

* 优点：`Robot` 角色切换稳定可用，失败率显著降低。
* 缺点：这不是“持久解锁”语义，重启后是否显示为已解锁不由该流程保证。

## 后续演进条件

如果未来要恢复“优先持久解锁”策略，需要先满足以下条件：

* 能稳定复现并确认原生解锁判定链路（不仅是单个 `GungeonFlags` 写入）。
* 在多个环境下验证解锁结果跨重启稳定生效。
* 保留失败兜底：即使持久解锁失败，仍可回退到强制切换，确保用户可用性。

## 适用范围

本决策当前仅覆盖 `Robot`。

`Bullet`、`Paradox`、`Gunslinger` 是否采用同策略，需基于各自稳定性和验证结果单独决策。
